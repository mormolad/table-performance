## 🎯 Цель Проекта

Продемонстрировать подход к разработке высоконагруженных интерфейсов для энтерпрайз-систем (CRM, биллинг, личные кабинеты).

**Ключевая задача:** Показать, как работать с миллионами записей без потери производительности, с заботой о ресурсах сервера и удобстве оператора.

## 🏗️ Архитектурные Решения

### 1. Почему Mock Generator вместо JSON-файла?
| Подход | Проблема | Наше решение |
|:---|:---|:---|
| Хранить 5 млн строк в JSON | Файл >500MB, долгая загрузка, раздувает репо | Генерация на лету по индексу |
| Загружать в память | Потребление >1GB RAM | Детерминированная генерация без хранения |
| Реальная БД | Сложность настройки для демо | Эмуляция задержки сети (300мс) |

**Код генератора:**
```javascript
// Данные не хранятся в памяти — генерируются по запросу? цель фронтенд показать
for (let i = start; i < end; i++) {
    data.push({
        id: i,
        name: `Subscriber_${i}`,
        tariff: TARIFFS[i % 4],
        balance: (Math.random() * 1000).toFixed(2)
    });
}
```

### 2. Почему AbortController?
В CRM операторы часто быстро меняют фильтры. Без отмены запросов:
- ❌ Сервер получает 10 запросов за 2 секунды
- ❌ Race condition: старый ответ может перезаписать новый
- ❌ Напрасная трата ресурсов бэкенда

**Наше решение:**
```typescript
useEffect(() => {
  if (controllerRef.current) {
    controllerRef.current.abort(); // Отменяем предыдущий
  }
  
  const controller = new AbortController();
  controllerRef.current = controller;
  
  fetch(url, { signal: controller.signal });
  
  return () => controller.abort(); // Очистка
}, [filters]);
```

### 3. Почему TanStack Virtual?
| Метрика | Без виртуализации | С виртуализацией |
|:---|:---|:---|
| DOM узлов | 5 000 000 | ~50-100 |
| Потребление памяти | >1GB | ~50MB |
| FPS при скролле | <10 | 55-60 |
| Время первой отрисовки | >30 сек | <1 сек |

**Настройки:**
- `estimateSize: 35` — высота строки
- `overscan: 5` — буфер для плавности
- `position: absolute` — рендеринг только видимой области

### 4. Почему Offset Pagination для демо?
Для продакшена рекомендуется **Cursor-based pagination**, но для демонстрации:
- ✅ Проще для понимания
- ✅ Легче реализовать фильтрацию
- ✅ Достаточно для показа концепции

## 🎯 Ключевые Фичи

| Фича | Реализация | Польза для бизнеса |
|:---|:---|:---|
| **Виртуализация** | TanStack Virtual | Работает с миллионами строк без лагов |

| **Обработка ошибок** | Error Boundaries + UI | Понятные сообщения, кнопка «Повторить» |

## 🚀 Production Ready: Что Добавить для Реального Продакшена

1.  **React Query / SWR** — кэширование, дедупликация запросов, background refetch
2.  **Cursor-based Pagination** — для избежания проблем с глубокими страницами
3.  **Debouncing фильтров** — 300-500мс задержка перед запросом
4.  **WebSocket** — для одновременного доступа к данным (например, когда меняют одновременно одну ячейку таблицы)
5.  **Feature Flags** — для постепенного rollout новых фич
6.  **Monitoring** — Sentry для ошибок, Performance API для метрик
7.  **Unit Tests** — Jest + React Testing Library
8.  **E2E Tests** — Playwright или Cypress
