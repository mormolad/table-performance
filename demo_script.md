# Скрипт демонстрации проекта (3 минуты)

## 1. Введение (30 сек)
- **Кто я:** [Ваше Имя], разработчик.
- **Что это:** Демонстрационный проект CRM-системы для работы с большими данными (5 000 000+ строк).
- **Проблема:** Обычные таблицы "умирают" на 1000+ строках. Моя задача — показать, как сделать интерфейс, который летает на миллионах.

## 2. Технический стек и Виртуализация (1 мин)
- **Стек:** React 18, TypeScript, TanStack Virtual.
- **Демонстрация скролла:** (Листаем таблицу). Обратите внимание на FPS — стабильные 60 кадров. 
- **Как это работает:** В DOM-дереве всегда только 20-30 строк, которые видны на экране. Остальные 4 999 970 строк существуют только как виртуальные координаты.
- **Память:** Потребление RAM фронтендом не растет при скролле и держится в районе 50MB.

## 3. Работа с сетью и AbortController (1 мин)
- **Проблема Race Condition:** Если оператор быстро переключает фильтры, старые запросы могут "догнать" новые и перезаписать данные.
- **Решение:** Использую `AbortController`. (Открываем вкладку Network, быстро меняем тарифы). 
- **Результат:** Видно, как предыдущие запросы получают статус `cancelled`. Это экономит ресурсы сервера и гарантирует актуальность данных.
- **Скелетоны:** При загрузке пользователь видит не пустой экран или спиннер, а структуру данных (Skeletons), что улучшает UX.

## 4. Ответы на вопросы (30 сек)
- **Почему не React Query?** 
  - *Ответ:* В реальном проекте я бы его использовал. Но здесь я хотел показать "подкапотную" работу с `AbortController`, `useEffect` и нативным `fetch/axios`, чтобы продемонстрировать понимание основ платформы и управления жизненным циклом запросов.
- **Почему Mock API?** 
  - *Ответ:* Чтобы не раздувать проект базой данных на 500MB. Генератор на бэкенде детерминирован — по одному и тому же индексу всегда возвращаются одни и те же данные.
